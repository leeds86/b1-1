# 실행 로그(대화 로그)

## 1. 시나리오 설명

| 항목 | 내용 |
| --- | --- |
| 테스트 목적 | 회의록 요약 AI '모이'의 실제 동작 검증 |
| 사용 모델 | Claude Sonnet 4.6 (웹)  |
| 프롬프트 버전 | v2 (페르소나 + Reasoning + Few-shot + Output Format + Constraints 통합본) |
| 입력 데이터 | 5인 참석 녹취록 (신규 앱 출시 / 고객센터 개선 / 예산 재배분, 약 1,200자) |
| 총 턴 수 | 13턴 |
| 테스트 유형 | 녹취록 입력 후 발언자 식별 ➔ 사실 기반 환각 및 수치 왜곡 검증 5회(존재하지 않는 안건, 잔액과 예비비 개념 혼동, 외부 지식 결합 차단 등) ➔ 실무 조건 변경(CFO 반려, QA 외주 견적가 변경)에 따른 동적 문맥 유지 검증 ➔ 문서 양식(포맷) 통제 및 원칙 충돌 시 AI의 역제안 수용 (공/사 격리 및 리스크 방어선 구상) ➔ 최종 임원 보고용 요약본 도출까지 총 13턴의 유기적인 비즈니스 시나리오 수행 |

## 3. 문제 지점과 개선 요약

- **[수치 왜곡 검증]**
    - **문제점:** 유저가 은근슬쩍 "인원 충원 결정된 거 맞지?" 하고 낚시 질문을 던졌을 때, 반박하지 못하고 그냥 "현재 8명, 충원 3명 필요합니다"라며 단순 팩트 나열에 그침 (수동적 요약)
    - **개선 방향:** 질문에 가짜 전제가 깔려 있으면, 첫 문장부터 **즉각 거절하는 단호함**을 심어줌
- **[잔액과 예비비 개념 혼동 검증]**
    - **문제점:** '예비비'라는 단어를 누가 말했는지는 기가 막히게 찾아냈지만, 정작 가장 중요한 "PM이 오늘 이 논의를 보류시켰다"는 결론(최종 상태)을 요약본에서 빼먹음
    - **개선 방향:** 단어 매칭보다 '안건별 최종 의사결정 상태'를 최우선 판정하는 추론 프로세스 보강
- **[외부 지식 결합 차단 검증]**
    - **문제점:** 환각 차단은 훌륭했으나 비교 데이터를 불릿으로 나열하여 가독성이 저하됨 (단호한 거절 톤 무뎌짐)
    - **개선 방향:** 비용·수치 대안 제시 시 '마크다운 표(Table) 양식 활용'을 `# Output Format` 규칙에 강제화
- **[문맥 유지 검증]**
    - **문제점:** '의견 개입 금지' 조항에 과도하게 묶여, 유저가 "임원 보고용이니까 '리스크'라는 무서운 단어 빼고 제목 예쁘게 바꿔줘"라고 검열을 요청하자 **실무진이 놓치면 안 되는 명확한 위험 요소가 될 수 있음**에도 이를 무기력하게 수용함
    - **개선 방향:** 시스템 무결성을 해치는 서식 변경 요청 시 거절을 보장하는 **예외 조항** 신설
- **[대안 선택 및 추가 수치 변경 요구]**
    - **문제점:** 마지노선(1,000만 원)을 초과한 예산 폭탄(1,200만 원)을 감지했음에도, 리스크 섹션에 단순 일정 질문만 남기는 등 **위기 상황을 과소평가함**
    - **개선 방향:** 예산이 초과하는 비상 상황이 터지면 경고 문구를 강제로 출력하도록 페르소나 압박 톤 고도화
- **[의견 개입 및 규칙 충돌 검증]**
    - **문제점:** 유저가 "수석 PM으로서 해결책 좀 제안해줘"라고 요구하자 규칙 방어에만 집중하여 수용하려는 **실무적 유연성**이 부족했음
    - **개선 방향:** Few-shot에 "의견 제안" 예시를 추가하거나 규칙에 "응답 의무 조항" 신설
- **[제안 수용 및 의견 개입 재요구]**
    - **문제점:** 규칙을 풀고 주관적 대안을 내보라고 하니, 회계 규정상 불가능한 '비용 다음 분기 이월'을 대안이랍시고 내놓거나 이미 상부에 고정된 출시 일정을 PM 맘대로 바꿀 수 있다고 과신하는 등 현업 감각이 떨어지는 엉성한 소리를 함
    - **개선 방향:** 제안을 출력하기 전에 스스로 검증하는 단계를 시스템 프롬프트에 넣거나 페르소나에 **할 수 있는 것과 없는 것**을 명시

## 4. 종합 결론: **'LLM 비서'에서 '지능형 에이전트'로의 도약 과제**

- v2 프롬프트는 팩트 추적 및 수치 무결성(Accuracy)은 완벽히 장착했으나, **비즈니스 리스크를 통제하는 방어적 판단력과 견제력**이 여전히 부재함
- 이번 실험의 개선 방향은 결국 AI에게 단순히 "요약하라"가 아니라, "프로젝트의 무결성을 지키기 위해 유저와 단호하게 밀당(트레이드오프)하라"는 **[방어적 판단력]의 기준 체계를 심어주는 것**
- 무료 모델의 경우, 추론 능력과 컨텍스트 윈도우(기억력의 범위)가 제한적이기 때문에 **Few-shot 예시를 보강하는 방법**과 **자기 검증 프롬프트를 추가**하여 스스로 점검하여 최종 출력을 하도록 개선
- 컨텍스트 윈도우 한계로 문맥이 끊기는 문제가 발생한다면, **LLM에게 자기 요약**(예, "매 10턴마다 대화 내용을 3줄로 요약하여
 답변 맨 앞에 [대화 요약]으로 표시하시오")을 지시하거나 시스템 프롬프트에 **핵심 변수**(변하지 않는 사실 정보)를 고정하는 방식으로 문맥을 유지
